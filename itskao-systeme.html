Infrastructure It’sKao

Le projet repose sur une stack Windows Server pilotée depuis C:\SERVEUR, avec des instances BeamMP, des scripts Lua mutualisés, une base centrale SQLite, un bot Discord, un backend FastAPI et une exposition web via nginx.

Windows Server BeamMP Lua SQLite Python FastAPI nginx

Vue d'ensemble

Le système n'est plus uniquement piloté par fichiers. L'état live des serveurs reste partagé sur disque, mais l'identité joueur, les sanctions, les reports et le temps de jeu sont maintenant centralisés.

Architecture

1. Instances BeamMP

Chaque serveur conserve sa configuration locale, ses logs et son player_info.json, mais charge sa logique Lua depuis des wrappers pointant vers SharedScripts.

2. Scripts mutualisés

Les modules security, gametime et report sont centralisés, ce qui évite la maintenance de dizaines de copies et fiabilise les mises à jour.

3. Base centrale

itskao_core.db est la source de vérité métier pour l'identité joueur, les sessions, les bans, les sanctions et les reports. Le pivot principal est désormais player_uuid.

4. Services applicatifs

Le bot Discord T11.py, les agrégateurs Python, le backend admin FastAPI et le site public partagent les mêmes données et la même logique de supervision.

Flux principaux

Le fonctionnement quotidien repose sur quelques chaînes de traitement très claires, ce qui rend l'exploitation plus lisible que le document source brut.

Flux
Etat live des serveurs Les instances BeamMP produisent player_info.json. L'agrégateur aggregate_servers.py consolide ensuite ces données dans all_servers_info.json, qui sert au site public, au backend admin et au bot Discord.
Identité et sécurité security/init.lua résout l'identité des joueurs, alimente la base centrale et pilote aussi la lecture des commandes distantes via les fichiers remote_commands_*.json.
Temps de jeu et reports gametime/init.lua écrit directement les sessions dans itskao_core.db, tandis que report/init.lua crée les reports et les réponses joueur dans cette même base.
Modération et administration T11.py lit la DB centrale, crée automatiquement des threads Discord par report et s'appuie sur la même source de vérité que l'interface admin FastAPI.

Exploitation quotidienne

Le projet montre une vraie logique d'administration système: orchestration, supervision, scripts de maintenance et distinction claire entre nominal et legacy.

Ops
  • Démarrage/arrêt: Demarrer-Tout.ps1 et Arreter-Tout.ps1 orchestrent la stack complète.
  • Writer nominal unique: aggregate_servers.py est le seul composant censé écrire all_servers_info.json.
  • Boucles critiques: supervision des reports, collecte de métriques, suivi des serveurs, publication Discord et contrôle des commandes distantes.
  • Compatibilité maîtrisée: les anciens fichiers comme banned_players.json, gametime.db ou Report\ sont conservés comme artefacts legacy, plus comme source principale.
